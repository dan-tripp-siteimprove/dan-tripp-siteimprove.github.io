This page demonstrates a way of "marking text as decorative". There are two parts to this "marking": a) exempting that text from contrast checks, and b) preventing that text from being announced by screen readers.

At the top of this page, there are breadcrumbs. Breadcrumbs separators (i.e. the forward slashes) are a common situation where page authors want to "mark text as decorative". The APG does this another way: by adding each forward slash not as a plain slash character, but rather as a 1px CSS vertical border which is then rotated 15 degrees to the right. This APG technique accomplishes (a) and (b) just fine. But I think that it's unnecessarily weird. Also, it limits the kind of separator characters that we can use. I might want to use double slashes, or bullet points, or emojis.

This page accomplishes (a) by adding the text via CSS pseudo-elements, rather than via straightforward text nodes in the DOM. Why is this necessary? Some people think that if you put aria-hidden on some text, that will "mark it as decorative" enough that the contrast checks won't check it. This is incorrect, because the contrast checks don't pay any attention to aria-hidden. The contrast checks are justified in doing this, because contrast is a visual thing, so the accessibility tree isn't involved in the user's experience, so (the argument goes) the accessibility tree shouldn't be involved in the checking of it.

This page (b) accomplishes with aria-hidden. No surprises here.

There is another common situation where page authors want to "mark text as decorative": background text AKA watermark text. Like if you have a word like "SAMPLE" in big letters (and poor contrast) lurking behind the regular context of your page. This demo page doesn't show that situation, but I think that this same technique would work for it.
